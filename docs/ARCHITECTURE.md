# ARCHITECTURE - 会议桌牌自动排序工具

## 1. 技术栈

| 层 | 选型 | 理由 |
|---|---|---|
| 交付形态 | 单文件 HTML（`index.html`） | 零安装、零部署、双击即用 |
| UI 渲染 | 原生 DOM + CSS 变量 | 无需框架，浏览器原生支持 |
| 平面图 | 原生 SVG（字符串拼接） | 矢量可编辑，可拖入 PPT/Word |
| 逻辑语言 | 纯 JavaScript（ES6） | 浏览器原生，无需编译 |
| 构建工具 | 无 | MVP 不引入任何构建步骤 |
| 依赖 | 零（无 npm / CDN / 外链） | 离线可用，U 盘拷贝即用 |
| 部署 | 无需部署 | 本地文件，可选放内网静态服务器 |

### 与设计稿的关系
- `design/` 是 React + Tailwind 的 UI 设计稿（v0.app 生成），为**视觉参考只读文件**
- 实际 `index.html` 复用设计稿的**色值、间距、字号、SVG 坐标**，但用原生 HTML/CSS/JS 重新实现
- 设计稿不包含座次排列算法（仅 mock 数据），算法在 `index.html` 中独立实现

## 4. 座次排列算法

### 输入/输出
```
输入：n（0–8 整数，该侧人数）
输出：长度为 8 的数组，索引 0→7 对应白板侧→短边侧
      有人座位存排名数字（1, 2, 3…），空位存 null
```

### 算法核心

**1 号位定位**：

同侧 8 个座位（索引 0–7，白板侧→短边侧），中心线在索引 3↔4 之间：

| 双方奇偶 | 客方 1 号位 | 主方 1 号位 | 说明 |
|---------|----------|----------|------|
| 均为偶数 | 索引 3 | 索引 4 | 1&2 各自居中（客 1@3/2@4，主 1@4/2@3），交叉相对 |
| 其余情况 | 索引 3 | 索引 3 | 双方 1 号同索引，保证正对 A[i]↔B[i] |

**填位顺序（中心→两侧，左右交替，左优先）**：

客方面门而坐 → 左 = 俯视图右侧 = 大索引方向。主方面白板而坐 → 左 = 俯视图左侧 = 小索引方向。双方"左手"方向相反，因此使用**不同的填位顺序**。

| 方 | 1 号在索引 3 | 1 号在索引 4 |
|---|-------------|-------------|
| 客方 | `[3, 4, 2, 5, 1, 6, 0, 7]` | `[4, 5, 3, 6, 2, 7, 1, 0]` |
| 主方 | `[3, 2, 4, 1, 5, 0, 6, 7]` | `[4, 3, 5, 2, 6, 1, 7, 0]` |

取填位顺序的前 n 个位置填入 1..n，其余位置填 null。

### 主客 1 号对坐保证

**对坐关系**：长边 A 索引 i ↔ 长边 B 索引 i（同索引同 x 坐标，直接正对，无需翻转或镜像）。

| 客方人数 | 主方人数 | 1 号对坐 | 机制 |
|---|---|---|---|
| 偶数 | 偶数 | ❌ 不需要 | 客 1@3/2@4，主 1@4/2@3，各自居中，交叉相对 |
| 奇数 | 奇数 | ✅ 需要 | 双方 1 号同索引 3，A[3]↔B[3] 自然正对 |
| 奇数 | 偶数 | ✅ 需要 | 双方 1 号同索引 3，A[3]↔B[3] 自然正对 |
| 偶数 | 奇数 | ✅ 需要 | 双方 1 号同索引 3，A[3]↔B[3] 自然正对 |

**验证**：

```
客 5（奇）· 主 5（奇）→ 非均偶，1 号同在索引 3：
  客方 [3,4,2,5,1] → [null,5,3,1,2,4,null,null]，1 号索引 3
  主方 [3,2,4,1,5] → [null,4,2,1,3,5,null,null]，1 号索引 3
  → A[3]↔B[3] 正对 ✓

客 6（偶）· 主 6（偶）→ 均偶，客 1@3 主 1@4：
  客方 [3,4,2,5,1,6] → [null,5,3,1,2,4,6,null]，1 号索引 3、2 号索引 4
  主方 [4,3,5,2,6,1] → [null,6,4,2,1,3,5,null]，1 号索引 4、2 号索引 3
  → 客 1↔主 2、客 2↔主 1 交叉相对，各自 1&2 居中 ✓

客 8（偶）· 主 8（偶）→ 均偶，客 1@3 主 1@4：
  客方 [3,4,2,5,1,6,0,7] → [7,5,3,1,2,4,6,8]，1 号索引 3、2 号索引 4
  主方 [4,3,5,2,6,1,7,0] → [8,6,4,2,1,3,5,7]，1 号索引 4、2 号索引 3
  → 满座，客 1↔主 2、客 2↔主 1 交叉相对，各自 1&2 居中 ✓
```

### 边界情况
| 输入 | 行为 |
|---|---|
| n = 0 | 返回全 null 数组，仅保留"长边 A·客方"标签 |
| n = 8 | 满座，全部 8 个位置都有数字 |
| n < 0 或 n > 8 | clamp 到 [0, 8]，console.warn 提示 |
| 非整数 | 向下取整 |

## 5. 数据模型

无后端、无数据库、无 LocalStorage。唯一的状态是用户输入的两个数字：

```javascript
// 页面运行时状态（仅存于内存，无持久化）
let state = {
  guestCount: 5,   // 客方人数，0–8
  hostCount: 5,    // 主方人数，0–8
}
```

## 6. 服务层约定

MVP 无 API 调用、无后端服务、无数据持久化。所有逻辑在浏览器内完成。

## 7. AI 引用机制

- 开发时 AI **必须同时读取** `docs/` 下所有 `.md` 文件（PRD / DESIGN / ARCHITECTURE）
- 修改代码前必须**确认不破坏** DESIGN.md 中的视觉规范（色值、间距、字号、SVG 坐标）
- 座次规则以 `桌牌摆放规则.txt` 为权威依据
- 布局以 `会议室布局.md` 为权威依据

## 8. 开发约束（禁止破坏）

- ❌ 禁止引入任何 npm 依赖 / CDN 外链（保持零依赖）
- ❌ 禁止引入构建工具（Webpack / Vite / etc）
- ❌ 禁止修改 `design/` 文件夹（设计稿只读参考）
- ❌ 禁止单文件超过 500 行（超过则考虑拆分逻辑到独立 .js 文件，但 MVP 阶段暂不引入）
- ❌ 禁止使用 ES Module（`type="module"`），保持最大兼容性

## 9. 验收标准

- [ ] 输入 0–8 任意客方/主方人数组合，平面图正确渲染
- [ ] 座位数字与排列规则一致（手动验证 5 组案例：5-5 / 3-3 / 8-8 / 4-6 / 0-2）
- [ ] 主客 1 号正对（双方 1 号在同一索引，物理位置垂直对齐）
- [ ] 空位显示虚线方块，有人座位显示实心方块 + 数字
- [ ] 下载 SVG 文件可拖入 PPT/Word 并正常显示
- [ ] 打印时隐藏输入区，仅保留平面图 + 图例
- [ ] 文件可在无网络环境下双击正常打开使用
